Running in circles


February 18, 2020

I have spent a few hours now looking at code and not actually writing much new. As mentioned previously, I realized that my game was getting fairly long for one file and that I should break it up into smaller sections as is best practice. I really thought this was going to be easier than it was, but a few hours later I haven’t managed to break it up much at all! Breaking it up initially presented with lots of problems untangling a spaghetti of classes, functions and variables I tried to import from one file to the next. This was the first time I had ever attempted doing this, and I thought it would be as simple as importing math, sys, random, or any other Python library. I was in for a surprise!

ImportError: cannot import name ‘Ship’ from partially initialized module 'cargo' (most likely due to a circular import)

This error message really puzzled me and had me searching through articles, Stack Overflow posts, and YouTube videos. Up until now, I really had no clue that circular dependencies were a bad thing. In my novice programmer brain, it seems reasonable to believe that I can send variables to one file and then back again so as to keep some sort of continuation. I am beginning to see how it can cause problems: the hard part is rewriting what has already been written to address this.

If you are reading this and haven’t yet experienced what I am talking about, this article seemed to help best in understanding the problems I had created for myself:

https://stackabuse.com/python-circular-imports/

I was able to break up my code into a few new files, but the majority remains as part of the main file it has always been a part of (pirate.py). Earlier today I had intended to break up most of the functions that make up the main menu of the game and its parts, to date that hasn’t been possible and may not even be necessary. Pirate.py is now 75 lines leaner than it was yesterday with the addition of 3 new files: cargo.py, ships.py, and character.py, each containing important classes pertaining to game function. I haven’t made as much progress today in terms of adding new game features, but I have started an important process in optimizing what I have written so far, cleaning it up, and hopefully making it easier on myself in the future. I have run around in circles learning about running around in circles, but today I have learned important lessons I won’t forget.

I’m really beginning to see and feel what John Ousterhout talks about in his book A Philosophy of Software Design. As I continue forward I want to remain mindful of overall design, keep an eye on dependencies, strive to write strategically instead of tactically, and take the time to make things easier on myself in the long-run.
